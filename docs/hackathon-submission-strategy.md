# Hackathon Submission Strategy

Community Spirit should enter hackathons as a distribution and feedback channel, not as the project's main identity.

The public project position stays:

```text
Community Spirit is a community life RPG and physical-AI social layer grounded in real spatial operations.
```

The hackathon framing can be:

```text
Community Spirit is the first reference implementation of CACP, a Community AI Collaboration Protocol for coordinating people, places, operators, AI agents, and future robots through community task contracts.
```

## What To Emphasize

- A working React/Vite demo, not only a concept deck.
- Synthetic community data with validation and no real residents, access control, cameras, payments, robots, or sensor logs.
- Community Pulse as the resident-facing loop: recommendation, route, join, check-in, XP, benefit, retention, and World Ops.
- CACP as the protocol layer: intent, permission, visibility, risk, resident touch, privacy boundary, fallback, feedback, and lifecycle.
- Task Contract View as proof that the protocol appears inside the product experience.
- Robot-ready export as a future integration surface, not a claim of robot control.

## Submission Copy

Short pitch:

```text
Community Spirit turns real community spaces into a low-pressure RPG layer for residents and an auditable task-contract layer for operators, AI agents, and future robots.
```

Technical pitch:

```text
The prototype uses synthetic POI, route, task, Pulse, retention, benefit, and contract data. Its CACP draft defines Community Task Contracts that describe who can act, where, under which permissions, with what privacy boundary, fallback owner, feedback signal, and lifecycle state before a task is handed to a person, AI agent, or robot.
```

Judge-safe boundary:

```text
This public demo is synthetic-only. It does not connect to real residents, real communities, access-control systems, cameras, payments, robots, or sensor logs.
```

## Event-window Honesty

If a competition requires work created during a specific event window, describe the event contribution as a scoped release on top of the existing open-source base:

- CACP draft release;
- Task Contract View;
- judge landing page;
- demo video refresh;
- new synthetic scenario;
- integration adapter;
- release polish and validation.

Do not claim the whole repository was created during the event window unless that is true.

## Recommended Submission Flow

1. Lead with the one-sentence positioning.
2. Show Community Pulse in the resident view.
3. Show check-in, XP, Spirit Points, and benefit conversion.
4. Open World Ops and show retention plus operator review.
5. Show Task Contract View and explain CACP.
6. Show robot-ready export as future handoff.
7. Close with the synthetic-data boundary and why CACP is different from MCP/A2A.
